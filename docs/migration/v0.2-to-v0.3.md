# Migration Guide: v0.2 → v0.3

A 5-minute checklist for upgrading from Machina v0.2.x to the latest v0.3.x
(currently **v0.3.1**).

!!! tip "Already on v0.3.0?"
    The breaking change below (typed `Capability`) landed in v0.3.0. If you are
    already on 0.3.0, skip to [v0.3.0 → v0.3.1](#v030-v031) for the deltas in
    the latest patch line (RAG upgrade + write-path safety).

## Breaking Changes

### 1. Capability type: `list[str]` → `frozenset[Capability]`

**What changed:** Connector capabilities are now a `frozenset[Capability]`
(typed enum) instead of `list[str]`.

**Migration:**

```python
# v0.2 (old)
class MyConnector:
    capabilities = ["read_assets", "create_work_order"]

# v0.3 (new)
from machina.connectors.base import Capability

class MyConnector:
    capabilities = frozenset({Capability.READ_ASSETS, Capability.CREATE_WORK_ORDER})
```

**Compatibility:** The connector registry accepts both formats during v0.3.x
(dual-accept). The old `list[str]` format will be removed in v0.4. Migrate now
to avoid warnings.

### 2. Capability values

The full `Capability` enum:

```python
# CMMS
READ_ASSETS, READ_WORK_ORDERS, GET_WORK_ORDER, CREATE_WORK_ORDER,
UPDATE_WORK_ORDER, CLOSE_WORK_ORDER, CANCEL_WORK_ORDER,
READ_SPARE_PARTS, READ_MAINTENANCE_PLANS, READ_MAINTENANCE_HISTORY

# IoT
SUBSCRIBE_TO_NODES, READ_NODE_VALUE, READ_NODE_VALUES, BROWSE_NODES,
SUBSCRIBE_TO_TOPICS, PUBLISH_MESSAGE,
GET_RELATED_READINGS, GET_LATEST_READING

# Documents
SEARCH_DOCUMENTS, RETRIEVE_SECTION

# Communication
SEND_MESSAGE, RECEIVE_MESSAGE

# Calendar
READ_CALENDAR_EVENTS, CREATE_CALENDAR_EVENT, DELETE_CALENDAR_EVENT
```

### 3. MCP server replaces the stub

The `machina.mcp.MCPServer` stub from v0.2.1 (which raised `NotImplementedError`)
is replaced by a real MCP server. If your code caught that error, remove the
try/except.

## New Features (No Migration Needed)

These are additive — existing code continues to work:

**Added in v0.3.0:**

- **MCP Server** — expose connectors via Model Context Protocol
  (`python -m machina.mcp`)
- **Excel/CSV Connector** — read/write maintenance data from spreadsheets
- **SQL Connector** — read from PostgreSQL, SQL Server, SQLite, DB2
- **GenericCmms YAML Mapper** — zero-Python entity mapping for REST APIs
- **ActionTracer v2** — cost tracking, conversation IDs, JSONL export with redaction
- **Docker deployment** — `docker compose up` with mock CMMS
- **systemd deployment** — production-ready service unit
- **Starter-kit template** — `odl-generator-from-text`

**Added in v0.3.1:**

- **RAG upgrade** for `DocumentStoreConnector` — hybrid retrieval (dense + BM25
  with RRF), cross-encoder reranking, layout-aware PDF/DOCX parsing (Docling),
  section-aware + parent-document chunking, structured citations, and a
  configurable embedder. All heavy deps are opt-in behind the new
  `[docs-rag-hybrid]` / `[docs-rag-rerank]` / `[docs-rag-parsing]` extras (or
  the aggregator `[docs-rag-pro]`); without them the connector degrades to the
  existing behaviour.
- **Write-path safety & idempotency** — deterministic content-hash work-order
  IDs (`auto_work_order_id`), layer-wide `@sandbox_aware` enforcement on every
  external-mutation path, durable atomic local persistence, and method-aware
  HTTP retry. See [Security → Sandbox Enforcement](../deployment/security.md#sandbox-enforcement-layer-wide).

## v0.3.0 → v0.3.1

No breaking API changes, but a few **observable behaviour changes** — review if
you depend on them:

- **`DocumentStoreConnector.search()` result shape.** `DocumentChunk.content`
  now carries the full **parent section** (after dedup-by-parent), not the small
  match passage that was embedded and ranked. If your code assumed `content` was
  a short snippet (e.g. fixed slicing), adjust it. New `DocumentChunk` fields
  `parent_id`, `start_offset`, `is_table` are appended (optional, positional
  construction preserved).
- **`AgentResponse.citations`** is a new field populated when the agent answers
  from documents. Code that constructs or pattern-matches `AgentResponse` by
  shape should account for it (it defaults to an empty list).
- **`predictive_pipeline` example now defaults to SANDBOX** (was LIVE). Pass
  `--live` to execute writes. `quickstart` keeps LIVE; every example now accepts
  `--sandbox` / `--live` symmetrically.
- **Excel/CSV `update_work_order` now persists to disk** when `write_mode` is
  set (previously CSV updates were cache-only and lost on restart).
- **Custom connectors using `request_with_retry`:** POST/PATCH are no longer
  retried on network/timeout errors or 503 by default (a timeout-after-success
  would duplicate the resource). Opt back in per call with
  `retry_on_network_error=True` for genuinely idempotent endpoints.

## Unreleased → next release

### Write confirmation now on by default (behaviour change)

**What changed:** `Agent(confirmations=...)` now defaults to `True` (a new
`confirmations:` YAML config key mirrors it). Mutating tool calls — e.g.
`create_work_order`, `execute_workflow` — now require a human confirmation
before they execute.

**Impact on programmatic callers:** with confirmations on and **no** `confirmer`
supplied, a write is fail-safe — it is **not** executed. Instead the agent
proposes the action and waits for a follow-up confirming message (the two-turn
degrade). If you drive the agent programmatically and expect autonomous writes:

```python
# Opt out of the confirmation gate (restores pre-default behaviour):
agent = Agent(..., confirmations=False)

# Or in machina.yaml:
# confirmations: false

# Or keep the gate and supply a synchronous confirmer:
async def confirmer(prompt: str) -> bool:
    ...  # render prompt, return the user's decision
await agent.handle_message_full(text, confirmer=confirmer, user_id="...")
```

`trigger_workflow` is a deliberate direct-execution path guarded by `sandbox`
only; it is **not** gated by `confirmations`.

**Known limitations:**

- **Anonymous senders cannot confirm (fail-safe withhold).** On a shared async
  channel, the two-turn confirmation binds the pending write to the sender's
  `user_id`. If the channel cannot supply a trusted sender identity (empty
  `user_id`), the write is **withheld** — not stored and not executed — so an
  unidentified participant can never confirm it.
- **Pending confirmations expire.** A pending write is only valid for the
  immediate follow-up; it expires after one hour (`_PENDING_ACTION_TTL_SECONDS`),
  so a much-later bare "ok" is treated as a fresh message, not a stale
  confirmation.
- **Email sender identity is NOT authenticated.** The email channel derives
  `chat_id` and `user_id` from the unauthenticated, spoofable `From:` header (no
  DKIM/SPF/allowlist). Two-turn write confirmation over email therefore provides
  **no integrity guarantee** — a spoofed sender could confirm a write. For
  write-confirmation use, gate the email connector behind sandbox mode or a
  verified-sender allowlist, or avoid confirmations over email.

## Checklist

- [ ] Update dependency: `pip install --upgrade "machina-ai[mcp]>=0.3.1"`
      (add `docs-rag` / `docs-rag-pro` if you want the upgraded RAG stack)
- [ ] If you have custom connectors: migrate `capabilities` from `list[str]` to
      `frozenset[Capability]`
- [ ] If you caught `MCPServer NotImplementedError`: remove the try/except
- [ ] If you read `DocumentChunk.content` as a short passage: it is now the full
      parent section — adjust any slicing (see [v0.3.0 → v0.3.1](#v030-v031))
- [ ] If you call `request_with_retry` for an idempotent POST/PATCH: pass
      `retry_on_network_error=True`
- [ ] Run your tests — `pytest` should catch any remaining issues
- [ ] (Optional) Try the MCP server: `python -m machina.mcp --config machina.yaml`
